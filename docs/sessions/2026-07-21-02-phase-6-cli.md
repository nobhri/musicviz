# Phase 6 CLI Retrospective

## Outcome

- Added the single Version 0 operation, `musicviz render project.yaml`.
- Moved the verified render implementation into a small `musicviz` package
  without changing the project schema or FFmpeg filtergraph.
- Registered the command as an installed project script and updated the locked
  uv environment for editable package installation.
- Added command-line tests for success, render errors, and invalid arguments.
- Completed Phase 6 and moved the roadmap to Phase 7 real-song validation.

## Decisions

- Used `argparse` from the standard library rather than adding a CLI dependency.
- Kept command dispatch separate from rendering so CLI behavior can be tested
  without invoking FFmpeg.
- Retained one command and one positional project path; no speculative options
  or additional subcommands were introduced.
- Explicitly listed the `musicviz` package for setuptools because ignored local
  `input/` and `output/` directories made flat-layout auto-discovery ambiguous.

## Verification

- Ran `uv sync` and confirmed the editable package and `musicviz` entry point
  install successfully.
- Ran `uv run pytest -q`; all 16 tests passed.
- Ran `uv run ruff check .` and `uv run ruff format --check .`; all checks
  passed.
- Rendered `output/phase-6-cli.mp4` with
  `uv run musicviz render project.yaml`.
- Confirmed H.264 video, AAC audio, 1080 x 1920, `yuv420p`, 30 fps, 150 video
  frames, and matching five-second stream and container durations.
- Played the complete output in ffplay and ran `git diff --check` successfully.

## Lessons

- Declaring a project script does not make uv install it when
  `tool.uv.package` is false; Phase 6 requires package installation explicitly.
- Setuptools auto-discovery treated ignored top-level media directories as
  possible packages. Explicit package configuration keeps the build scope
  small and predictable.
- Editable setuptools installation creates `*.egg-info`; this generated
  artifact belongs in `.gitignore`.
- Real project text exposed a publishing limitation: the fixed Arial fonts do
  not contain Japanese glyphs. YAML loading and temporary text files preserve
  UTF-8 correctly, so the issue is font coverage rather than text decoding.
  The limitation is recorded in the active documentation for Phase 7.

## Next step

Render three real song projects without code changes. Select and verify a
Japanese-capable fixed font if Japanese titles or artist names are required,
then address only recurring publishing problems demonstrated by those renders.
