# Phase 0 Baseline Retrospective

Date: 2026-07-20

## What we accomplished

- Completed the direct FFmpeg baseline at 1080 × 1920 and 30 fps with H.264
  video, AAC audio, and `yuv420p` output.
- Matched the video, audio, and container durations exactly and verified the
  complete output with both `ffprobe` and `ffplay`.
- Replaced the 76.5-second local MP3 with a reproducible five-second, 44.1 kHz,
  stereo PCM WAV generated by FFmpeg.
- Replaced the sparse portrait artwork with a reproducible 1600 × 1600 RGB
  `testsrc2` PNG that makes crop, scale, zoom, and color problems easier to
  identify.
- Updated the project status, roadmap, and render-spike evidence. Phase 1 slow
  zoom is now the current roadmap phase.

## Decisions

- Use one short WAV fixture during direct pipeline development. Its exact
  timing avoids lossy-codec delay and keeps render checks fast.
- Keep MP3 support in Version 0, but add a separate MP3 fixture only when input
  format validation or automated integration tests require it.
- Use deterministic FFmpeg-generated test media instead of committing local
  media or maintaining duplicate fixtures.
- Keep the input duration explicit in the direct spike command. The later
  Python wrapper must obtain it from the input rather than hard-code it.

## What we learned

- `-shortest` did not guarantee equal stream durations for a looped image and
  MP3: libx264 had already accepted enough buffered frames to extend the video
  to 78.27 seconds.
- Disabling encoder latency removed that overrun but made the video 0.10
  seconds shorter than the audio. Supplying the probed duration with `-t`
  produced exact 30 fps frame counts and matching stream durations.
- FFmpeg's `sine` and `testsrc2` sources provide small, repeatable fixtures for
  audio and artwork pipeline checks.
- A square, high-contrast test pattern represents typical album artwork and is
  more diagnostic for vertical cropping and slow zoom than sparse portrait
  artwork.

## Verification

- Generated and probed the five-second PCM WAV.
- Rendered `output/phase-0-baseline.mp4` from the generated WAV and PNG.
- Confirmed 1080 × 1920, 30 fps, H.264, AAC, `yuv420p`, 150 video frames, and
  matching 5.000-second video, audio, and container durations with `ffprobe`.
- Played the complete output with `ffplay` and checked crop, color, audio, and
  clean termination.
- Ran `git diff --check`.

## Friction and limitations

- The generated files under `input/` and `output/` are intentionally ignored,
  so a fresh checkout must run the documented fixture-generation commands.
- The direct command still contains an input-specific `-t 5`; automatic
  duration discovery belongs in the later Python wrapper.
- The fixture validates the pipeline but does not replace real-song validation
  for MP3 inputs, artwork aesthetics, or publishing quality.
